# 砍价活动「砍价项可配置化」详细需求与设计说明

## 1. 背景与目标

当前砍价活动前台页面的转盘落点（每一档砍价金额）由代码硬编码维护，运营无法在活动管理后台灵活调整，存在以下问题：

- 调整配置需要研发改代码 + 发版，响应慢；
- 不同砍价活动（感恩节/音乐会）无法独立配置落点概率；
- 抽奖策略无法精细化运营（只改金额不能改概率）。

本次改造目标：

1. 将前台砍价项抽离为活动级可配置数据（写入 `bargainSettings`）；
2. 保持砍价项数量固定（7项），默认值与现有前台一致；
3. 管理后台支持同时配置「砍价金额 + 概率」；
4. 概率采用单项百分比输入，且总和必须等于 100%；
5. 配置区在后台页面排版紧凑、整齐、易读。

---

## 2. 需求范围

### 2.1 功能范围

- 活动管理后台（新增/编辑活动）
  - 当活动类型为 `bargain` 时，显示砍价项配置区；
  - 固定渲染 7 行砍价项；
  - 每行可编辑：`amount`（砍价金额）与 `probability`（概率%）。

- 管理端保存逻辑
  - 提交前校验所有项概率总和必须为 100；
  - 校验不通过时阻止提交并提示。

- 管理云函数
  - 接收并标准化 `bargainItems`；
  - 对缺省配置使用默认值补齐；
  - 对非法概率总和直接拒绝写入。

- 活动云函数（前台抽奖）
  - 读取活动配置中的 `bargainItems`；
  - 按概率进行加权随机命中；
  - 保留 `segments` 输出用于前台转盘展示兼容。

### 2.2 非功能要求

- 向后兼容历史活动（未配置时回退默认）；
- 不改变前台页面已有 UI 交互；
- 不引入额外集合与迁移脚本。

---

## 3. 默认数据与固定项定义

固定项数量：`7`

默认砍价项（沿用现有前台逻辑）：

1. 金额 `120`，概率 `14%`
2. 金额 `180`，概率 `14%`
3. 金额 `200`，概率 `14%`
4. 金额 `260`，概率 `14%`
5. 金额 `320`，概率 `14%`
6. 金额 `500`，概率 `14%`
7. 金额 `0`，概率 `16%`

概率总和：`100%`

---

## 4. 数据结构设计

`activity.bargainSettings` 新增字段：

```json
{
  "startPrice": 1500,
  "floorPrice": 998,
  "shareRewardAttempts": 1,
  "ticketingMode": "paid-ticket",
  "bargainItems": [
    { "amount": 120, "probability": 14 },
    { "amount": 180, "probability": 14 },
    { "amount": 200, "probability": 14 },
    { "amount": 260, "probability": 14 },
    { "amount": 320, "probability": 14 },
    { "amount": 500, "probability": 14 },
    { "amount": 0, "probability": 16 }
  ]
}
```

约束：

- `bargainItems` 长度固定按 7 项处理（后端以默认模板补齐/截断）；
- `amount >= 0` 且取整；
- `probability >= 0` 且取整；
- `sum(probability) === 100`。

---

## 5. 交互与校验设计

### 5.1 后台编辑器交互

- 仅当 `activityType === 'bargain'` 时展示配置区；
- 使用三列表格式：序号 / 砍价金额 / 概率(%)；
- 每行均可直接输入数字；
- 配置区标题明确提示“概率总和100%”。

### 5.2 校验策略

前端校验：

- 点击保存时先计算 7 项概率总和；
- 若不为 100，toast 提示并阻止提交。

后端校验：

- `normalizeBargainSettings` 再次计算总和；
- 若不为 100，直接抛错拒绝入库。

说明：前后端双校验可避免绕过前端提交非法配置。

---

## 6. 抽奖算法设计

### 6.1 方案

采用加权随机：

1. 计算总权重 `total = Σ probability`；
2. 生成随机值 `r in [0, total)`；
3. 顺序扣减每项概率，`r < 0` 时命中该项；
4. 返回命中项金额作为本次砍价值。

### 6.2 与前台展示的关系

- 前台转盘展示继续依赖 `segments`；
- 后端将 `segments` 由 `bargainItems.amount` 同步映射，保证展示与抽奖来源一致。

---

## 7. 兼容性与容错

- 新活动：未显式配置 `bargainItems` 时自动使用默认 7 项；
- 老活动：读取时若配置缺失或异常，使用默认项兜底；
- 运行时保护：仅当读取到的概率总和为 100 时覆盖默认配置，避免异常配置影响线上抽奖。

## 8. 涉及代码模块

- 管理云函数：`cloudfunctions/admin/index.js`
  - `normalizeBargainSettings` 增加 `bargainItems` 标准化与概率校验。

- 活动云函数：`cloudfunctions/activities/index.js`
  - 默认配置增加 `bargainItems`；
  - 新增砍价项标准化与按概率抽取函数；
  - `spinBhkBargain` 使用概率模型命中金额。

- 管理端页面：`miniprogram/subpackages/admin/activities/index.{js,wxml,wxss}`
  - 表单新增 `bargainItems` 字段；
  - 新增输入处理与保存前校验；
  - 新增紧凑表格样式。

---

## 9. 验收标准（UAT）

1. 进入活动管理，活动类型选择砍价活动后，出现 7 行砍价项配置区；
2. 修改任一行金额/概率后可保存；
3. 当概率总和不为 100 时，前端阻止提交并给出提示；
4. 若绕过前端直接调用接口提交非法概率，总后端拒绝写入；
5. 保存合法配置后，前台砍价活动可正常抽奖，且展示转盘档位与配置一致；
6. 配置区在常见机型下显示紧凑整齐，无明显错位。

---

## 10. 发布与回滚

### 发布步骤

1. 部署 `cloudfunctions/admin`；
2. 部署 `cloudfunctions/activities`；
3. 上传并发布小程序管理端代码包。

### 回滚策略

- 若发现异常，可先回滚云函数到上一版本；
- 数据层面保留 `bargainItems` 字段不影响旧逻辑（旧版本忽略该字段）。
